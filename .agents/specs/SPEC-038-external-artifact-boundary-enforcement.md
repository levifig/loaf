# SPEC-038: External artifact boundary enforcement

## Problem Statement

Loaf-produced local artifacts under `.agents/` are internal scaffolding. They are useful while planning, coordinating, reviewing, and resuming work, but they are not stable public references.

When internal artifact IDs or paths leak into commits, branches, pull requests, changelogs, public docs, GitHub, or Linear, they create brittle provenance:

- dead links when `.agents/` files move, archive, or remain local-only
- noisy public history tied to implementation process instead of shipped outcome
- confusing dual identity such as `SPEC-123` plus `ENG-456`
- pressure to preserve disposable files because external artifacts reference them

Loaf already has partial guidance against this, but enforcement is incomplete and examples still leak internal IDs.

## Strategic Alignment

- **Vision:** External project history should describe what shipped, not how Loaf internally coordinated it.
- **Personas:** Team leads need clean public surfaces; solo developers need private scaffolding without long-term cleanup tax.
- **Architecture:** Hooks validate boundary violations. CLI commands generate compliant payloads where Loaf owns the workflow.

## Solution Direction

Establish a hard artifact-boundary rule:

> `.agents/` artifacts are private scaffolding. External surfaces must describe outcomes, never internal Loaf artifact IDs, paths, tracks, phases, or local provenance.

Ban these from external surfaces:

- `SPEC-*`
- `TASK-*`
- `.agents/...`
- `Track N`
- `Phase N`
- local spec/task/session/report file paths
- Loaf-only status vocabulary when it is not meaningful outside Loaf

Allowed external references:

- External durable system IDs such as Linear issue IDs or GitHub issue/PR numbers.
- Magic words using external IDs, for example `Closes ENG-456`.
- Human outcome language, for example `Harden release changelog preservation`.

Internal Loaf surfaces may still use internal IDs. That includes specs, sessions, local task metadata, reports, generated internal review packets, and the future work ledger from SPEC-039.

## Relationship To Other Specs

- **SPEC-037** defines specs as mutable internal work definitions, which makes this boundary necessary.
- **SPEC-039** provides the backend-neutral mapping that lets external systems use their own IDs without exposing Loaf internal IDs.

## Scope

### In Scope

- Audit existing hooks and classify them by purpose, surface, failure mode, and usefulness.
- Update git workflow guidance for commits, branch names, PR titles, PR bodies, changelogs, and release notes.
- Update `/implement` examples that currently include `SPEC-*` in commit subjects or public closeout text.
- Add or strengthen validators for:
  - commit subject and body
  - branch names
  - PR titles and descriptions
  - changelog/release-note text
  - generated Linear/GitHub payloads where Loaf CLI creates them
- Make enforcement configurable only where false positives are plausible; default should block obvious leaks.
- Document local-mode uncertainty: even without Linear, public artifacts should avoid Loaf internal IDs because the future ledger can preserve mappings internally.

### Out of Scope

- Creating or syncing Linear issues. That belongs to SPEC-039.
- Redefining spec semantics. That belongs to SPEC-037.
- Building hook metrics storage. This spec should leave hooks structured so future SQLite metrics can record firings, warnings, blocks, overrides, and false-positive outcomes.
- Retroactively rewriting existing git history or historical public artifacts.

### Rabbit Holes

- Allowing HTML comments in PR bodies for internal IDs. Resist initially: hidden public text is still public leakage and tends to be copied.
- Building a perfect natural-language classifier for every possible leak. Resist: start with deterministic patterns.
- Enforcing local-only file references in all docs. Resist: docs may legitimately mention repo paths; the target is Loaf internal artifacts and IDs in external/public surfaces.

### No-Gos

- No `SPEC-*` or `TASK-*` in commit subjects.
- No `SPEC-*` or `TASK-*` in branch names.
- No `SPEC-*`, `TASK-*`, `.agents/...`, `Track N`, or `Phase N` in PR titles.
- No Loaf internal IDs or `.agents/` paths in Linear issue titles/descriptions generated by Loaf.
- No internal artifact IDs in changelog or release notes.

## Hook Audit Model

Each hook should be classified in source documentation or an audit report:

```text
purpose: enforce | warn | inject-context | obsolete
surface: commit | push | pr | tool | session | release
failure_mode: block | warn | silent
false_positive_risk: low | medium | high
decision: keep | change | remove
metrics_ready: yes | no
```

Future SQLite-backed metrics can record:

- hook fired
- hook warned
- hook blocked
- override used
- runtime
- rule ID
- false-positive resolution

This spec only prepares the taxonomy and validation behavior.

## Risks

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| Validators block legitimate external IDs that resemble Loaf IDs | Medium | Medium | Scope regexes to known internal prefixes and path patterns |
| PR bodies need some internal trace for review | Medium | Medium | Use Loaf-generated internal reports and local ledger, not public PR text |
| Hooks become noisy and get bypassed | Medium | High | Audit hooks, remove obsolete checks, make messages precise |
| Linear mode still leaks via model-authored text | Medium | High | SPEC-039 moves Linear payload generation into CLI commands |

## Open Questions

- [ ] Should local-only projects ever allow `TASK-*` in commit bodies, or should the ban be universal? Recommended default: universal ban.
- [ ] Should branch validators run as hooks, CLI checks, or both?
- [ ] Should hook enforcement be blocking immediately, or warn first for one release?
- [ ] Which existing hooks are obsolete once CLI owns Linear payload generation?

## Test Conditions

- [ ] Commit validator rejects internal artifact IDs and `.agents/` paths in subjects.
- [ ] Commit validator rejects or warns on internal artifact IDs and `.agents/` paths in bodies, according to chosen policy.
- [ ] Branch-name validation rejects internal IDs and artifact names.
- [ ] PR-format validation rejects internal IDs, `.agents/` paths, `Track N`, and `Phase N` in title and description.
- [ ] Changelog/release-note validation rejects internal artifact IDs.
- [ ] Existing source examples are rewritten to use outcome language.
- [ ] Hook audit report exists with keep/change/remove decisions.
- [ ] Tests cover allowed external IDs such as `ENG-456` magic words separately from banned Loaf IDs.

## Priority Order

1. **Track A - Policy and examples.** Update guidance and remove known bad examples. Go/no-go: no source guidance tells users to expose Loaf internal IDs externally.
2. **Track B - Validators.** Add deterministic checks for commits, branches, PR text, and changelog text. Go/no-go: tests prove banned Loaf IDs fail and allowed external IDs pass.
3. **Track C - Hook audit.** Review existing hooks and classify keep/change/remove. Go/no-go: each relevant hook has an explicit decision and false-positive risk.
