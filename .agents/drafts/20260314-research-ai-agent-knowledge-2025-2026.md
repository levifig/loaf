# Knowledge Management for AI Coding Agents: State of the Art (2025-2026)

Research compiled March 2026.

## 1. How Major AI Coding Tools Handle Project Knowledge

### Claude Code

**Instruction files:** `CLAUDE.md` -- the primary mechanism. Hierarchical loading from multiple locations:
- `~/.config/claude/CLAUDE.md` -- global, cross-project instructions
- Project root `CLAUDE.md` -- committed to git, shared with team
- `.claude/CLAUDE.md` -- project-level (also git-committed)
- `CLAUDE.local.md` -- personal overrides (gitignored)
- `.claude/rules/` -- path-scoped rules (glob-matched, similar to Cursor)
- Parent directory `CLAUDE.md` files -- inherited up the tree
- Supports `@import` syntax for pulling in other files

**Auto-Memory (MEMORY.md):** Shipped enabled by default as of v2.1.32 (Feb 2026). Claude writes its own learnings to `~/.claude/projects/{project}/memory/MEMORY.md`. Key distinction: CLAUDE.md is *your* instructions to Claude; MEMORY.md is *Claude's notebook* about your project. First 200 lines of MEMORY.md load every session; topic files load on demand. Machine-local, not committed to git.

**Skills system:** Directory-based, following the [Agent Skills open standard](https://agentskills.io). Each skill is a directory with a `SKILL.md` entrypoint (YAML frontmatter + markdown content). Skills become `/slash-commands`. Claude auto-discovers skills from `.claude/skills/` directories. Description quality drives auto-invocation accuracy -- the description is what Claude uses to decide when to load a skill. Character budget for skill descriptions scales at ~2% of context window (fallback: 16,000 chars). Skills can have sidecar files (`.claude-code.yaml`) for Claude-specific fields like `user-invocable`, `context: fork`, `agent`, etc.

**Adoption stat:** As of early 2026, only ~35% of Claude Code users configure a structured CLAUDE.md, despite it being the single biggest productivity lever.

Sources:
- [Claude Code Memory Docs](https://code.claude.com/docs/en/memory)
- [SFEIR Deep Dive on CLAUDE.md](https://institute.sfeir.com/en/claude-code/claude-code-memory-system-claude-md/deep-dive/)

- [Inside Claude Code Skills (Mikhail Shilkov)](https://mikhail.io/2025/10/claude-code-skills/)

### Cursor

**Rules system:** Migrated from legacy `.cursorrules` (single file, deprecated) to `.cursor/rules/` directory with individual `.mdc` files. Each rule file has YAML frontmatter with three fields:
- `description` -- what the rule covers (used for "Apply Intelligently" mode)
- `globs` -- file patterns (Minimatch/glob syntax, e.g. `**/*.ts`, `apps/web/**`)
- `alwaysApply` -- boolean

**Rule activation modes:**
1. **Always** -- `alwaysApply: true`, loaded every conversation
2. **Glob-matched** -- activated when files matching the glob are referenced
3. **Agent-selected** -- `alwaysApply: false` with no glob; agent reads description and decides relevance

**Known limitations:** Context window recency bias causes rules to be ignored late in long conversations (model prioritizes recent messages over system-injected rules). Stale globs fail silently when paths are renamed. No built-in staleness detection.

**Global/user-level rules:** As of early 2026, Cursor supports User Rules in settings, but true cross-project global rules (like a `~/.cursor/rules/` directory) remain a community feature request, not a shipped feature.

Sources:

- [Cursor Rules for AI (legacy docs)](https://docs.cursor.com/context/rules-for-ai)
- [DataCamp: Cursor Rules Tutorial](https://www.datacamp.com/tutorial/cursor-rules)
- [Cursor Forum: Global Rules Request](https://forum.cursor.com/t/global-cursor-rules-directory/50049)

### Windsurf (Codeium)

**Rules:** `.windsurfrules` file -- permanent context layer, loaded every session for the project. Global rules loaded first, then project-level.

**Memory system (Cascade Memories):** Autogenerated structured records stored locally in `~/.codeium/windsurf/memories/`. Not just notes -- they include user stories, architectural decisions, process changes, technical standards, troubleshooting steps. Cascade's assembly pipeline loads rules first, then memories, then open files, then codebase retrieval, then recent actions before assembling the final prompt.

**How it differs:** Windsurf's memories are more structured than Claude's auto-memory -- they're typed records rather than freeform markdown. However, they're workspace-scoped and local, similar to Claude's approach.

Sources:
- [Windsurf Cascade Memories Docs](https://docs.windsurf.com/windsurf/cascade/memories)
- [Windsurf Rules & Workflows (Paul Duvall)](https://www.paulmduvall.com/using-windsurf-rules-workflows-and-memories/)
- [Windsurf Flow Context Engine](https://markaicode.com/windsurf-flow-context-engine/)

### Aider

**Configuration:** `.aider.conf.yml` -- loaded hierarchically from home directory, git repo root, and current directory (later files win on conflicts).

**Conventions:** `CONVENTIONS.md` loaded via `read: CONVENTIONS.md` in config. Model-agnostic -- conventions work across any LLM backend. Can load multiple files: `read: [CONVENTIONS.md, anotherfile.txt]`.

**Repository map:** Aider's distinguishing feature. Generates a concise map of the entire git repo including key classes, functions, types, and call signatures. Uses a graph ranking algorithm where source files are nodes and edges represent dependencies. The map is optimized to fit the active token budget by selecting only the most relevant portions via graph ranking. This is the closest thing to automated codebase understanding -- no manual configuration needed.

**Approach difference:** Aider leans on automated code analysis (repo map) rather than human-authored instructions. The conventions file is optional supplementary context, not the primary knowledge mechanism.

Sources:

- [Aider YAML Config](https://aider.chat/docs/config/aider_conf.html)

### OpenAI Codex

**AGENTS.md:** The primary instruction mechanism, now an open standard under the AAIF. Codex walks from the project root down to the current working directory, checking for `AGENTS.override.md`, then `AGENTS.md`, then fallback filenames, building an instruction chain.

**Best practice that emerged:** Treat AGENTS.md as a ~100-line table of contents / map, with the actual knowledge living in a structured `docs/` directory. AGENTS.md points to deeper sources of truth rather than being the encyclopedia itself.

**Nested files:** Subfolders can have their own AGENTS.md for per-component instructions. OpenAI's own repos use dozens of nested AGENTS.md files.

**Skills integration:** Codex supports Agent Skills (the open standard). Skills can be installed and used as repeatable workflows for things like verification, release prep, PR review.

**Adoption:** Over 60,000 open-source projects had adopted AGENTS.md by late 2025. Over 20,000 GitHub projects adopted the format.

Sources:

- [Codex Agent Skills](https://developers.openai.com/codex/skills/)
- [OpenAI Blog: Skills and Agents SDK](https://developers.openai.com/blog/skills-agents-sdk/)

### Google Gemini CLI

**GEMINI.md:** Hierarchical loading, similar to Claude Code:
- `~/.gemini/GEMINI.md` -- global, all-project instructions
- Project root and parent directories (up to `.git` boundary)
- Subdirectories (respecting `.gitignore` and `.geminiignore`)

**Advanced features:**
- `@file.md` import syntax for breaking large files into components
- `GEMINI_SYSTEM_MD` environment variable to source core instructions from an external file
- `system.md` in `.gemini/` directory for behavioral overrides

**Approach:** Closest to Claude Code's hierarchical model. Notable for the `@import` syntax and the `.geminiignore` for controlling what subdirectory instructions get loaded.

Sources:
- [Gemini CLI GEMINI.md Docs](https://google-gemini.github.io/gemini-cli/docs/cli/gemini-md.html)
- [Structured approach to bloated GEMINI.md (Google Cloud blog)](https://medium.com/google-cloud/practical-gemini-cli-structured-approach-to-bloated-gemini-md-360d8a5c7487)

### GitHub Copilot

**Instruction files:**
- `.github/copilot-instructions.md` -- repository-level custom instructions (GA)
- `.instructions.md` files -- path-specific instructions with YAML frontmatter (shipped July 2025)
- Organization-level instructions -- lowest priority in the hierarchy
- Personal instructions -- highest priority

**Priority hierarchy:** Personal > Repository > Organization. All sets of relevant instructions are provided to Copilot (they merge, not override).

**Recent additions (2025-2026):**
- Agent Mode (GA April 2025) -- multi-file editing, test execution, verification
- Multiple `.instructions.md` support (July 2025) -- path-specific via YAML frontmatter
- Copilot Spaces (GA Sept 2025) -- integrated management of code, docs, and notes
- Agent-specific instructions (Nov 2025) -- `excludeAgent` property to control which agents see which rules
- Code Review integration -- `copilot-instructions.md` now supported in code review context

**Agent Skills support:** Copilot adopted the Agent Skills open standard. VS Code docs describe how to use Agent Skills with Copilot.

Sources:
- [GitHub Copilot Custom Instructions Docs](https://docs.github.com/copilot/customizing-copilot/adding-custom-instructions-for-github-copilot)
- [VS Code Copilot Custom Instructions](https://code.visualstudio.com/docs/copilot/customization/custom-instructions)
- [VS Code Agent Skills](https://code.visualstudio.com/docs/copilot/customization/agent-skills)

### Zed

**Rules system:** Rules Library interface with full editor, syntax highlighting. Rules can be set as defaults (loaded every session). Supports:
- `@` mentions for files, directories, symbols, previous threads, rules files, diagnostics
- `/project` slash command for quick context injection
- Explicit context control -- no automatic codebase indexing; you choose what goes in

**Approach difference:** Zed is intentionally minimalist about automatic context. Rather than indexing everything, it gives explicit control over what enters context. This is a philosophical counterpoint to tools like Aider (which auto-analyze) or Windsurf (which auto-load memories).

Sources:
- [Zed AI Rules Docs](https://zed.dev/docs/ai/rules)
- [Zed AI Overview](https://zed.dev/docs/ai/overview)

---

### Roo Code / Cline / Kilo Code

**Roo Code:** Multi-agent, role-driven VS Code extension. Organizes into modes (Architect, Code, Debug, Ask, Custom), each with specific tool subsets. Notable for the "Memory Bank" pattern -- community-developed persistent memory files for context-aware workflows. Multiple variations exist: `cursor-memory-bank`, `roocode-memorybank-optimized`, `RooFlow`, etc.

**Cline:** Model-agnostic, stepwise planning. The OG with 5M+ installs.

**Kilo Code:** Fork raising $8M seed (Dec 2025). Competing on reliability.

**Common pattern:** All three load `.clinerules` or similar project-level instruction files. The memory bank pattern emerged from the community, not the tools themselves.

Sources:

## 2. How Tools Handle USER Knowledge (Personal, Cross-Project)

### The Landscape

| Tool | Global/User Config Location | Cross-Project? | Git-Committed? |
|------|---------------------------|----------------|----------------|
| Claude Code | `~/.config/claude/CLAUDE.md` | Yes | No (user-local) |
| Claude Code | `~/.config/claude/PROFILE.md` | Yes | No (user-local) |
| Claude Code | `~/.claude/projects/{project}/memory/MEMORY.md` | Per-project | No |
| Cursor | User Rules (in settings) | Yes | No |
| Windsurf | Global `.windsurfrules` | Yes | No |
| Gemini CLI | `~/.gemini/GEMINI.md` | Yes | No |
| Aider | `~/.aider.conf.yml` | Yes | No |
| Copilot | Personal instructions | Yes | No |
| Zed | Default rules in Rules Library | Yes | No |

### Claude Code's Approach (Most Developed)

Claude Code has the most layered personal knowledge system:
1. **`~/.config/claude/CLAUDE.md`** -- Global instructions for all projects (coding style, communication preferences, tool preferences)
2. **`~/.config/claude/PROFILE.md`** -- Personal background and context (who you are, what you work on, your goals)
3. **Auto-memory per project** -- Claude's own learnings, scoped to each git repo
4. **`CLAUDE.local.md`** -- Per-project personal overrides (gitignored)

### The Gap

No tool currently maintains a true "user knowledge graph" that persists semantic understanding of a user across projects. The closest is Claude Code's auto-memory + PROFILE.md combination, but it's still file-based and manually curated (PROFILE.md) plus per-project auto-notes (MEMORY.md). There's no cross-project learning -- what Claude learns about you in Project A doesn't transfer to Project B unless you write it in your global CLAUDE.md.

**Mem0's OpenMemory** is the most ambitious attempt to fill this gap -- an MCP-based memory server that works across coding agents (Cursor, VS Code, Claude, etc.) and persists user patterns, preferences, and setup. But it's an external add-on, not built into any tool.

### Cursor's Gap

Cursor's lack of a global rules directory is a notable gap. Community requests date back to at least mid-2025, but as of March 2026, project-level `.cursor/rules/` remains the only shipped mechanism. User Rules exist in settings but are limited compared to the file-based rule system.

## 3. Agent Memory Frameworks and Research

### Letta (formerly MemGPT)

**Architecture:** OS-inspired tiered memory:
- **Core memory** -- always in-context, embedded in system instructions. The "working memory."
- **Archival memory** -- general read-write datastore, overflow for core memory. Agent can search and retrieve.
- **Recall memory** -- conversation history, retrievable on demand.

**Key insight:** Agents actively manage their own memory -- deciding what to keep in-context vs. what to store externally. This is the closest analog to human memory management.

**2025-2026 status:** Letta Code is #1 model-agnostic open-source agent on Terminal-Bench. Conversations API (Jan 2026) enables shared memory across parallel user experiences. Letta Evals framework (Oct 2025) for testing stateful agents.

Sources:
- [Letta Docs](https://docs.letta.com/concepts/memgpt/)
- [Letta Benchmarking Blog](https://www.letta.com/blog/benchmarking-ai-agent-memory)

---

### LangChain / LangGraph

**Memory types:**
- **Short-term (thread-scoped):** State persisted via checkpointer to a database. Resumed by thread ID.
- **Long-term (cross-session):** Stored via LangGraph Store, shared across threads. Supports semantic, episodic, and procedural memory types.

**LangMem SDK:** Dedicated toolkit for extracting and managing procedural, episodic, and semantic memories. Integrates natively with LangGraph.

**Production features (LangGraph 1.0):** Durable state with automatic persistence. Built-in save/resume at any point without custom DB logic.

**Integration:** MongoDB Store for LangGraph enables cross-session memory with flexible filtering and semantic search.

Sources:
- [LangGraph Memory Docs](https://docs.langchain.com/oss/python/langgraph/memory)
- [LangMem SDK Launch](https://blog.langchain.com/langmem-sdk-launch/)
- [LangChain/LangGraph 1.0](https://blog.langchain.com/langchain-langgraph-1dot0/)

---

### CrewAI

**Memory system:**
- Short-term: ChromaDB with RAG for recent interactions
- Long-term: Persistent across sessions with LLM-analyzed content (scope, categories, importance scoring)
- **MemorySlice:** A view across multiple, possibly disjoint scopes -- recall from several branches simultaneously

**Cognitive memory:** LLM analyzes content when saving, infers scope/categories/importance. Adaptive-depth recall with composite scoring (semantic similarity + recency + importance).

**Key pattern:** CrewAI Flows that run periodically don't start from scratch -- they recall previous findings, identify deltas, and maintain a "living knowledge base" that refines over cycles.

Sources:
- [CrewAI Memory Docs](https://docs.crewai.com/en/concepts/memory)
- [CrewAI Cognitive Memory Blog](https://blog.crewai.com/how-we-built-cognitive-memory-for-agentic-systems/)

### Microsoft Agent Framework (AutoGen successor)

**Status:** AutoGen and Semantic Kernel are in maintenance mode. Agent Framework is the successor, targeting 1.0 GA by end of Q1 2026.

**Memory:** Persistent storage and telemetry by default (Azure Cosmos DB, Application Insights, Fabric). Long-term memory in Foundry Agent Service stores chat summaries, user preferences, task outcomes.

**Checkpointing:** Built-in workflow checkpointing -- pause, persist, resume from any checkpoint. Automatic persistence with no manual state management.

Sources:
- [Microsoft Agent Framework Overview](https://learn.microsoft.com/en-us/agent-framework/overview/)
- [Agent Framework Migration Guide](https://learn.microsoft.com/en-us/agent-framework/migration-guide/from-autogen/)

### Mem0

**Position:** Universal memory layer, works across LLM providers and agent frameworks.

**Architecture:** Two-phase pipeline (Extraction + Update). Enhanced variant Mem0g adds graph-based store for multi-session relationships.

**Performance claims:** 26% accuracy boost, 91% lower p95 latency, 90% token savings on LOCOMO benchmark.

**OpenMemory:** MCP-based memory server specifically for coding agents. Works with Cursor, VS Code, JetBrains, Claude, OpenAI, LangGraph, LlamaIndex. Stores coding patterns, preferences, setup across tools.

**Funding:** $24M raised October 2025.

### Academic Research (2024-2025)

**Key papers:**
- **A-MEM (Feb 2025):** Agentic memory using Zettelkasten method -- dynamic indexing and linking to create interconnected knowledge networks. Agents organize their own memories rather than using flat storage.
- **"Memory in the Age of AI Agents" survey:** Comprehensive paper list at [github.com/Shichun-Liu/Agent-Memory-Paper-List](https://github.com/Shichun-Liu/Agent-Memory-Paper-List)
- **HiAgent:** Hierarchical working memory management for agents
- **Zep:** Temporal Knowledge Graph architecture for agent memory
- **"How Memory Management Impacts LLM Agents" (May 2025):** Empirical study of experience-following behavior

**Memory structure taxonomy:** Chunks, knowledge triples, atomic facts, summaries, or mixed memory -- each with different tradeoffs between recall precision, contextual unity, and reasoning exactness.

**Storage paradigms:** Cumulative, reflective/summarized, purely textual, parametric, and structured approaches.

**Open problems:** Memory overload from naive strategies, lack of human-like forgetting mechanisms, inadequate preference-weighted recall.

## 4. The Knowledge Drift Problem

### The Core Issue

AI agent instructions go stale. Paths change, dependencies upgrade, conventions evolve, but instruction files don't update themselves. This manifests as:

- **Silent glob failures:** Cursor rules with renamed paths match nothing, with no warning
- **Architectural drift:** Agents make locally sensible but globally inconsistent decisions when instructions don't reflect current architecture
- **Stale conventions:** Instructions reference deprecated patterns, old library versions, or removed APIs
- **Context window recency bias:** Even when correct instructions exist, models prioritize recent conversation content over system-injected rules in long sessions

### What's Being Tried

**Spec-driven development:** Treat specifications as contracts between humans and agents. The spec is the source of truth that agents continuously refer to, reducing drift. This is the most widely recommended approach (Addy Osmani's "The 80% Problem", Anthropic's Agentic Coding Trends Report).

**Instruction capsules:** Short, focused instruction files scoped to specific contexts (Cursor's glob-matched rules, Copilot's path-specific `.instructions.md` files). The theory: smaller, more focused instructions are easier to maintain and less likely to drift.

**Auto-memory as partial solution:** Claude Code's MEMORY.md and Windsurf's Cascade Memories capture learnings automatically, but these capture *agent observations*, not authoritative project truth. They can drift too.

**What's NOT working:** No tool automatically detects stale instructions. No tool compares instruction files against actual code to find mismatches. No CI/CD integration validates that CLAUDE.md or AGENTS.md matches the current codebase. This remains a fully manual process.

### Industry Observations

From the [Anthropic 2026 Agentic Coding Trends Report](https://resources.anthropic.com/hubfs/2026%20Agentic%20Coding%20Trends%20Report.pdf): AI makes more mistakes when it lacks business rules and architectural constraints. Providing repo-specific instruction capsules and configuration schemas reduces misconfigurations and logic drift.

From [VentureBeat](https://venturebeat.com/ai/why-ai-coding-agents-arent-production-ready-brittle-context-windows-broken): Brittle context windows and broken refactors are cited as reasons agents aren't production-ready. Missing operational awareness is the root cause.

From [Addy Osmani](https://addyo.substack.com/p/the-80-problem-in-agentic-coding): The "80% problem" -- agents get 80% right but the last 20% requires exactly the project-specific knowledge that tends to drift.

## 5. Cross-Project and Shared Knowledge Patterns

### Monorepo Patterns

**Nested instruction files:** All major tools support this. AGENTS.md, CLAUDE.md, GEMINI.md, and Cursor rules can all be nested in subdirectories for per-component instructions. OpenAI's own repos demonstrate this pattern with dozens of nested AGENTS.md files.

**Nx Polygraph:** Available for Nx Enterprise users (H1 2025). Connects multiple repositories into a federated meta-repo. AI tools benefit from the unified dependency graph and project structure that Nx provides.

**Context-Driven Development pattern:** Structure the monorepo so that AI agents can navigate it effectively -- clear module boundaries, consistent naming, documentation as code. The architecture itself becomes the knowledge.

### Shared Knowledge / Federation

**Agent Skills as shared packages:** The Agent Skills ecosystem is the closest thing to "shared knowledge across repos." Skills are installed from registries (skills.sh, SkillsMP) and provide reusable domain knowledge. Companies like Vercel, Prisma, Supabase, Stripe ship official skills that any project can install.

**Organization-level instructions:**
- GitHub Copilot: Organization-level custom instructions (lowest priority, merged with repo and personal)
- Claude Code: Managed policy files for organizations
- Cursor: No organization-level feature as of March 2026

**What doesn't exist yet:** True federated knowledge -- where updating a convention in one repo automatically propagates to all repos that reference it. Every tool treats each repo as an island. The closest workaround is git submodules or symlinks to shared instruction files, which is fragile.

### Standardization Efforts

**AAIF (Agentic AI Foundation):** Formed December 2025 under the Linux Foundation. Founding members: Anthropic (contributed MCP), OpenAI (contributed AGENTS.md), Block (contributed goose). Platinum members include AWS, Google, Microsoft, Cloudflare, Bloomberg. Mission: open standards for agentic AI to avoid fragmentation and vendor lock-in.

**Agent Skills specification:** Published by Anthropic December 2025 as an open standard at [agentskills.io](https://agentskills.io). Adopted by Microsoft, OpenAI, Atlassian, Figma, Cursor, GitHub. Compatible with Claude Code, Cursor, GitHub Copilot, Goose, Codex CLI, Windsurf, Gemini CLI, Roo Code, Trae, and more.

**AGENTS.md as open standard:** Released by OpenAI August 2025, contributed to AAIF. Over 60,000 projects adopted by late 2025.

**Skill marketplaces:**
- **skills.sh** (Vercel, Jan 2026): Primary registry, 57,000+ indexed skills
- **SkillsMP** (community): 351,000+ skills indexed via GitHub crawling
- **skillmarketplace.ai**: Another competing marketplace
- Growth: The ecosystem crossed 350K skills by early 2026, barely two months after first marketplaces launched

Sources:

## Summary: What's Actually Working vs. What's Theoretical

### Shipped and Working (Production-Ready)

| Pattern | Status | Tools |
|---------|--------|-------|
| Hierarchical instruction files | Mature | All tools |
| Glob-matched rule activation | Mature | Cursor, Copilot, Claude Code |
| Automated repo mapping | Mature | Aider |
| Agent Skills (open standard) | Shipping | Claude Code, Copilot, Codex, Cursor, Gemini CLI, Windsurf, Zed |
| Auto-memory (agent-written) | Shipping | Claude Code, Windsurf |
| Skill marketplaces | Early but growing fast | skills.sh, SkillsMP |
| Cross-session persistence | Shipping (framework-level) | LangGraph, Letta, CrewAI, Mem0 |

### Partially Working / Early Stage

| Pattern | Status | Notes |
|---------|--------|-------|
| Global/user-level instructions | Fragmented | Claude Code and Gemini CLI do it well; Cursor and Copilot are limited |
| Organization-level instructions | Early | Copilot has it; others don't |
| MCP-based universal memory | Early | Mem0 OpenMemory exists but adoption unclear |
| Federated cross-repo knowledge | Barely exists | Nx Polygraph for Nx users; nothing general |

### Not Solved / Theoretical

| Problem | Status |
|---------|--------|
| Automatic staleness detection for instructions | **No tool does this** |
| CI/CD validation of instruction files | **No tool does this** |
| Cross-project learning (user knowledge transfer) | **No tool does this natively** |
| Self-updating project knowledge | **Theoretical only** |
| Instruction file diffing against code changes | **No tool does this** |
| Human-like memory forgetting/prioritization | **Academic research only** |

### Key Takeaways for Loaf

1. **The Agent Skills standard won.** Anthropic published it, and everyone adopted it. Loaf is already building on this -- that's the right bet.

2. **Hierarchical instruction loading is table stakes.** Every tool does it. The differentiation is in *what* you put in those files and how you structure the knowledge.

3. **Auto-memory is the new frontier.** Claude Code and Windsurf ship it. Others will follow. The question is whether agent-written memory can be trusted as authoritative or just supplementary.

4. **Knowledge drift is the unsolved problem.** Nobody has automated detection or maintenance. The best current practice is small, focused, scoped instruction files (Cursor's approach) combined with spec-driven development.

5. **Cross-project knowledge sharing is the biggest gap.** Skills marketplaces are the closest solution -- package knowledge as reusable skills. But personal/user knowledge that transfers across projects is basically unsolved except for simple global instruction files.

6. **Mem0 is worth watching.** Their MCP-based universal memory layer (OpenMemory) is the most ambitious attempt at cross-tool, cross-project memory. $24M in funding suggests they'll keep pushing.

7. **The AAIF standardization umbrella matters.** Having Anthropic, OpenAI, Google, Microsoft, and AWS all in the same foundation means the standards (MCP, AGENTS.md, Agent Skills) have real staying power.
